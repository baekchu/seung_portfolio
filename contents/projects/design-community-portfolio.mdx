## 프로젝트 개요

**Asterpiece**는 디자이너·일러스트레이터·작가 등 다양한 창작자들이 작품을 공유하고, 공모전에 참여하며, 서로 소통할 수 있는 **종합 창작 커뮤니티 플랫폼**입니다.

단순 포트폴리오 업로드 사이트를 넘어 시각 작품(캔버스), 글·소설(서재), 커뮤니티 게시판(놀이터), 공모전 캘린더, 개인 워크스페이스까지 창작 활동 전반을 아우릅니다. Firebase를 백엔드 인프라로 활용하여 별도 서버 없이 실시간 데이터 동기화를 구현했습니다.

## 핵심 기능 구성

### 🎨 캔버스 (Canvas) — 시각 작품 포트폴리오

디자이너와 일러스트레이터가 이미지 작품을 업로드하고 탐색하는 공간입니다. 12개 카테고리로 분류하여 원하는 장르를 필터링해 볼 수 있습니다.

| 카테고리 |
|---|
| 일러스트 / 사진 / 애니메이션 / 캐릭터 / 카툰 |
| 제품 디자인 / 타이포그래피 / 웹 디자인 / 모바일 디자인 |
| 브랜딩·로고 / 마케팅 디자인 / 기타 |

- Firebase Storage에 이미지(webp/gif/png/jpg) 업로드 → CDN 자동 서빙
- 작품 카드에 좋아요, 스크랩, 댓글 기능
- 작성자 프로필로 바로 이동 가능
- `next/image`로 이미지 최적화 및 lazy loading 적용


### 📚 서재 (Library) — 글·소설 창작 공간

텍스트 기반 창작자를 위한 공간입니다. 시, 소설, 웹소설, 희곡, 웹툰 대본 등 다양한 글 장르를 지원합니다.

| 카테고리 |
|---|
| 글귀 / 소설 / 창작 / 아이디어 / 수필 / 시 / 일기 |
| 평론 / 칼럼 / 인터뷰 / 리뷰 / 논픽션 / 동화 |
| 희곡 / 극본 / 라디오극 / 웹소설 / 웹툰 / 대본 / 만화대본 / 기타 |

### 🏆 공모전 (Contest) — 공모전 캘린더

공모전 일정을 캘린더 형태로 확인하고 참여할 수 있습니다. 월별 보기에서 날짜를 선택하면 해당 일의 공모전 목록을 조회합니다.


### 🎲 놀이터 (Bulletin) — 커뮤니티 게시판

창작자들이 자유롭게 의견을 나누고 아이디어를 교류하는 커뮤니티 공간입니다.

### 🖥️ 워크스페이스 (Workspace)

로그인한 사용자의 개인 작업 공간. 업로드한 작품 관리, 진행 중인 프로젝트 정리에 활용합니다.

### 👤 프로필 페이지

각 사용자의 개인 포트폴리오 페이지가 자동으로 생성됩니다. 업로드한 캔버스·서재 작품이 프로필에 집약되어 자신만의 온라인 포트폴리오로 기능합니다.

## 6단계 온보딩 플로우

첫 방문자에게 서비스 주요 기능을 순서대로 안내하는 스텝 온보딩을 구현했습니다.

```
Step 1: 환영 — "아스터피스에 오신 것을 환영합니다"
Step 2: 캔버스 소개
Step 3: 서재 소개
Step 4: 놀이터 소개
Step 5: 공모전 소개
Step 6: 워크스페이스 소개
```

온보딩 완료 여부는 localStorage에 저장하여 재방문 시 표시하지 않도록 처리했습니다.

## 기술 스택

| 기술 | 선택 이유 |
|---|---|
| **Next.js (App Router)** | 이미지 최적화(`next/image`), 파일 기반 라우팅, Vercel 배포 최적화 |
| **TypeScript** | Firestore 문서 구조에 대한 타입 안정성, 대규모 코드베이스 유지보수성 |
| **Firebase Firestore** | 실시간 구독(`onSnapshot`), 복잡한 쿼리 없이 NoSQL 유연한 데이터 모델링 |
| **Firebase Auth** | Google OAuth 포함 다중 인증 방식, 사용자 세션 관리 |
| **Firebase Storage** | 이미지·GIF·webp 직접 업로드 + CDN 자동 배포, 별도 서버 불필요 |
| **Zustand** | Redux 없이 전역 상태 단순화, Context 리렌더 최소화 |
| **TailwindCSS** | 빠른 UI 구현, 다크모드·반응형 레이아웃 일관성 |

## 데이터 구조

Firestore 컬렉션은 기능 단위로 분리했습니다.

```
/users/{userId}
  ├── displayName, photoURL, bio
  └── bookmarks: string[]        // 스크랩한 작품 ID 목록

/canvas/{artworkId}
  ├── title, category, authorId
  ├── images: string[]           // Storage URL 배열 (다중 이미지 지원)
  ├── likeCount, viewCount
  └── /comments/{commentId}
        └── text, authorId, createdAt

/library/{postId}
  ├── title, genre, content, authorId
  └── /comments/{commentId}

/contests/{contestId}
  └── title, deadline, url, description

/bulletin/{postId}
  └── title, content, authorId, createdAt
```

## 상태 관리

Zustand 스토어를 기능 단위로 분리하여 전역 상태를 관리했습니다.

```typescript
// stores/authStore.ts
interface AuthStore {
  user: FirebaseUser | null;
  setUser: (user: FirebaseUser | null) => void;
}

// stores/canvasStore.ts
interface CanvasStore {
  artworks: Artwork[];
  category: string;
  setArtworks: (list: Artwork[]) => void;
  setCategory: (cat: string) => void;
}
```

`onAuthStateChanged`로 로그인 상태를 감지하여 `authStore`에 저장하고, 각 페이지 컴포넌트는 직접 스토어를 구독합니다. Context API 대신 Zustand를 쓴 이유는 중첩 컴포넌트 리렌더를 줄이기 위해서입니다.

## Firebase 보안 규칙

인증 여부와 작성자 일치 여부로 읽기/쓰기 권한을 분리했습니다.

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /canvas/{artworkId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth.uid == resource.data.authorId;
    }
    match /canvas/{artworkId}/comments/{commentId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow delete: if request.auth.uid == resource.data.authorId;
    }
    match /users/{userId} {
      allow read: if true;
      allow write: if request.auth.uid == userId;
    }
  }
}
```

## 트러블슈팅

### onSnapshot 메모리 누수

`onSnapshot` 리스너를 `useEffect`에서 등록하고 cleanup 없이 방치하면 컴포넌트 언마운트 후에도 리스너가 살아있어 메모리 누수가 발생했습니다.

```typescript
useEffect(() => {
  const q = query(collection(db, "canvas"), orderBy("createdAt", "desc"));
  const unsub = onSnapshot(q, (snap) => {
    setArtworks(snap.docs.map((d) => ({ id: d.id, ...d.data() } as Artwork)));
  });
  return () => unsub(); // cleanup
}, []);
```

### GIF 업로드 용량 초과

사용자가 대용량 GIF를 업로드할 때 Firebase Storage 업로드가 타임아웃되는 문제가 있었습니다. 클라이언트 측에서 `file.size`를 체크하여 10MB 초과 시 업로드를 막고 안내 메시지를 표시하는 방식으로 처리했습니다.

```typescript
const MAX_SIZE = 10 * 1024 * 1024; // 10MB
if (file.size > MAX_SIZE) {
  alert("파일 크기는 10MB를 초과할 수 없습니다.");
  return;
}
```

### 카테고리 필터와 실시간 구독 충돌

카테고리를 변경할 때마다 새로운 Firestore 쿼리로 `onSnapshot`을 재구독해야 했는데, 이전 구독을 해제하지 않으면 데이터가 중복으로 쌓였습니다. `category` 상태가 바뀔 때마다 이전 `unsubscribe`를 호출하고 새 리스너를 등록하는 패턴으로 해결했습니다.

```typescript
useEffect(() => {
  const q = category === "전체"
    ? query(collection(db, "canvas"), orderBy("createdAt", "desc"))
    : query(collection(db, "canvas"), where("category", "==", category), orderBy("createdAt", "desc"));
  const unsub = onSnapshot(q, (snap) => {
    setArtworks(snap.docs.map((d) => ({ id: d.id, ...d.data() } as Artwork)));
  });
  return () => unsub();
}, [category]);
```

### Vercel 배포 후 Firebase 인증 도메인 오류

로컬에서는 정상 동작하던 Google 로그인이 Vercel 배포 후 `auth/unauthorized-domain` 오류가 발생했습니다. Firebase Console → Authentication → Settings → Authorized domains에 Vercel 배포 도메인(`asterpiece.com`, `*.vercel.app`)을 추가하여 해결했습니다.

### Firebase Storage CORS 오류

Firebase Storage에서 직접 이미지 URL을 참조할 때 일부 브라우저에서 CORS 오류가 발생했습니다. Storage 버킷 CORS 설정 파일(`cors.json`)을 작성하고 `gsutil` 명령으로 적용하여 해결했습니다.

```json
[
  {
    "origin": ["https://www.asterpiece.com"],
    "method": ["GET"],
    "maxAgeSeconds": 3600
  }
]
```
