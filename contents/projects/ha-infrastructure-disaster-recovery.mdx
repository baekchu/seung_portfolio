## 프로젝트 개요

서울 본사·대전 지사·부산 지사 3개 거점과 DMZ 구간을 Frame Relay WAN으로 묶고, 여기에 OSPF 멀티 Area, EIGRP 재분배, HSRP 이중화, VPN, DMZ 기반 서비스 분리까지 얹은 재해복구 인프라를 팀 프로젝트로 설계하고 구축했습니다.

이 중에서 **저는 대전 지사와 부산 지사 서버 구축을 전부 맡았습니다.**

| 항목 | 내용 |
|---|---|
| 주관기관 | 한국정보교육원 |
| 수행기간 | 2026.03 (2026.03.03 ~ 2026.03.31) |
| 개발 형태 | 팀 프로젝트 (대전·부산 지사 서버 구축 담당) |

전체 인프라는 크게 5개 영역으로 나눠집니다.

| 영역 | 구성 | 주요 기술 |
|---|---|---|
| 서울 본사 | OSPF Area 1, RIPv2 내부 재분배 | HSRP, PAT, VPN, SSH, ACL |
| DMZ | 인터넷 공개 서비스 구간 | Mail, Samba(전사파일), HAProxy, Web/DB |
| OSPF Backbone | Area 0, Frame Relay WAN | Multi Point / Point to Point, ABR 3개 |
| 대전 지사 | OSPF Area 4 | PAT, VPN, SSH, ABR(Area 3/4) Virtual Link |
| 부산 지사 | OSPF Area 2 + EIGRP | HSRP, PAT, VPN, ACL, Ether Channel |

서울 본사는 OSPF Area 1로 운영되는데, 내부 일부 구간이 RIPv2로 잡혀 있어서 OSPF와 재분배를 걸어줘야 했습니다. 서버실은 시험망(서버실 1)과 공용망(서버실 2)으로 나눠서, 테스트하다 잘못 건드린 트래픽이 운영 쪽으로 넘어가지 않게 분리해뒀습니다. 서버실 2에는 WEBDB1M, WEBDB2, WEBDB3을 올려 웹 서비스를 돌립니다.

DMZ는 인터넷과 내부망 사이에 두는 완충 구간입니다. Mail 서버, Samba(전사 파일 보관용), HAProxy 로드밸런서, Web1·Web2·WebDB를 여기에 배치했습니다. HAProxy가 Web1/Web2로 트래픽을 나눠주고, WebDB는 Slave로 붙여서 읽기 부하를 받습니다. DMZ 자체를 내부망과 완전히 다른 세그먼트로 떼어놨기 때문에, 혹시 DMZ 서버가 뚫리더라도 내부망으로 바로 들어오지는 못합니다.

WAN 구간(OSPF Area 0)은 Frame Relay의 Multi Point와 Point to Point를 섞어서 구성했습니다. ABR 3대(Area 0/1, Area 0/2, Area 0/3)가 각 거점 Area와 Backbone을 이어주고, Internet World 연결도 Area 0에서 받습니다. SSH는 포트 5506으로 바꾸고 ACL을 같이 걸어서 관리 트래픽을 제한했습니다.

## 담당 역할 — 서버 구축

### 대전 지사 서버실 (Windows Server)

대전 지사 서버실은 Windows Server 환경으로 구성했습니다.

![대전 지사 네트워크 구성도](/images/projects/ha-infra-daejeon.png)

#### 네트워크 구성

| VLAN | 서브넷 | 부서 |
|---|---|---|
| VLAN 10 | 192.168.200.0/25 | 서버실 (Windows Server) |
| VLAN 20 | 192.168.200.128/25 | 통합관제팀 |
| VLAN 30 | 192.168.210.0/25 | 보안연구팀 |
| VLAN 40 | 192.168.210.128/25 | 지사행정팀 |
| VLAN 50 | 192.168.220.0/25 | 유출조사팀 |

L2 스위치는 VTP Server/Client로 묶어서, VLAN 정보를 Server에서 한 번만 잡으면 나머지 스위치로 알아서 퍼지게 했습니다. 스위치마다 일일이 VLAN을 손으로 넣지 않아도 돼서 편했습니다.

#### 서버 구성 상세

ATC 관제 서버는 각 부서 VLAN에서 올라오는 트래픽과 네트워크 이벤트를 실시간으로 모아 모니터링합니다. 모은 데이터는 관제 DB에 쌓아두고, 이상 징후를 보거나 나중에 추적할 때 들여다봅니다.

DJ_DHCP 서버는 VLAN마다 scope를 따로 잡아서 각 부서 클라이언트에 맞는 서브넷 주소를 뿌려줍니다. 서버나 프린터처럼 IP가 고정이어야 하는 장비는 MAC 기반 Reservation으로 묶어 IP가 겹치지 않게 했습니다.

관제 DB는 ATC 서버의 로그·이벤트를 저장하는데, localhost랑 ATC 서버 IP에서만 붙을 수 있게 바인딩을 막아놨습니다.

#### OSPF Area 4 — Virtual Link

대전 지사는 Area 4인데, 하필 Area 0(Backbone)과 직접 붙어있지 않은 비연속 Area 구조였습니다. OSPF에서는 모든 Area가 Backbone과 직접 연결돼야 하는 게 원칙이라, ABR에 Virtual Link를 걸어 Area 3을 전송 Area로 빌려 쓰는 식으로 Backbone과 논리적으로 이어줬습니다. 이걸 안 잡으면 Area 4 경로가 전체 라우팅 테이블에 안 뿌려져서 지사 간 통신 자체가 안 됐습니다.

#### DB 이중화 — RAID + Master-Slave 복제

재해복구 인프라에서 결국 제일 중요한 건 데이터를 지키는 일이라고 봤습니다. 그래서 디스크 레벨과 DB 레벨, 두 군데서 각각 이중화를 걸어줬습니다.

**1단계 — RAID 1+0 (디스크 이중화)**

먼저 RAID 1+0(RAID 10)으로 잡았습니다. RAID 1(미러링)로 묶은 디스크 쌍을 다시 RAID 0(스트라이핑)으로 묶는 구조인데, 단순 RAID 1보다 읽기가 빠르면서도 미러 쌍에서 한 디스크가 죽어도 데이터는 살아있습니다. DB 서버는 읽기·쓰기가 워낙 잦다 보니 성능도 가용성도 포기하기 싫어서 이 구성을 골랐습니다. 디스크를 갈아끼우면 자동 리빌드로 알아서 정상으로 돌아옵니다.

**2단계 — MariaDB Master-Slave 복제 (DB 이중화)**

RAID로 디스크는 지킨다 해도, DB 프로세스 자체가 죽어버리면 서비스는 그대로 멈춥니다. 그 경우까지 대비하려고 MariaDB Master-Slave 복제를 추가로 걸었습니다.

| 구분 | 역할 | 상세 |
|---|---|---|
| Master DB (GW3 DB) | 쓰기 전담 | INSERT·UPDATE·DELETE 처리, Binary Log(binlog)를 Slave에 실시간 전송 |
| Slave DB | 읽기 + 복제 대기 | Relay Log를 통해 Master 상태를 동기화, 장애 시 Master로 수동 승격 |

복제가 잘 돌아가는지는 `SHOW SLAVE STATUS\G`로 확인했습니다. `Slave_IO_Running`·`Slave_SQL_Running`이 둘 다 `Yes`인지, `Seconds_Behind_Master`가 0 근처로 붙는지를 종종 들여다봤습니다. Master가 죽었을 때를 대비해서, Slave에서 `STOP SLAVE` → `RESET SLAVE ALL` → 서비스 연결 전환 순으로 승격 절차도 미리 정리해뒀습니다.

#### Samba 서버 — 크로스 플랫폼 파일 공유 및 DB 연동 자동화

부산 지사 서버실은 Rocky Linux 서버랑 Windows 클라이언트가 같이 쓰는 환경입니다. 그래서 Samba(SMB/CIFS)를 올려서, OS가 뭐든 같은 공유 디렉터리로 접근할 수 있게 했습니다.

여기서 그냥 파일 공유만 하기는 아까워서 DB 백업 자동화까지 엮었습니다.

- **자동 백업**: cron으로 `mariadb-dump`를 주기적으로 돌려서, 덤프 파일을 Samba 공유 디렉터리(`/srv/samba/backup`)에 타임스탬프 붙은 이름으로 자동 적재
- **크로스 플랫폼 접근**: 백업 파일이 공유 디렉터리에 떨어지면, 바로 Windows 관리자 PC에서 네트워크 드라이브로 열어서 복구에 쓸 수 있음
- **자동 마운트**: `/etc/fstab`에 `cifs` 타입으로 등록해둬서, 서버를 재부팅해도 공유 디렉터리가 알아서 다시 붙음

#### 권한 관리 — 계층적 접근 제어

**SSH 접근 제어**

- `sshd_config`에서 `PasswordAuthentication no` 설정으로 패스워드 로그인을 완전히 비활성화하고 **공개키 인증**만 허용
- `PermitRootLogin no`로 root 직접 로그인 차단, 관리 작업은 일반 계정 로그인 후 `sudo`로만 수행
- `AllowUsers`로 접속 가능한 계정을 명시적으로 지정, 미등록 계정은 인증 단계 이전에 차단

**Samba 공유 권한**

- `smb.conf`에서 공유 섹션별로 `valid users`를 지정해 인가된 계정만 접근 가능
- 팀별 공유 디렉터리에 `read only = yes` / `writable = yes`를 구분 적용
- `map to guest = never`로 익명 접근을 원천 차단
- Samba 권한만 믿지 않고 Linux 파일 시스템 권한(chmod 770, chown)도 같이 걸어서, Samba를 우회해 들어와도 막히게 함

**DB 계정 권한 분리**

- 애플리케이션 서비스 계정: `GRANT SELECT ON db.* TO 'app_user'@'내부IP'` — 읽기 전용
- DBA 계정: 스키마 변경·계정 관리 전용, 서버 콘솔 또는 내부 관리망에서만 접근 허용
- `GRANT` 시 `'user'@'%'` 대신 `'user'@'192.168.100.x'` 형태로 접근 허용 IP를 명시
- `bind-address`를 localhost 및 내부 서버 IP로 제한하여 외부에서 MariaDB 포트(3306) 직접 접근 차단

## 전체 인프라 구성 (팀 공동)

### 라우팅 — OSPF + EIGRP 재분배

| 거점 | 프로토콜 | 비고 |
|---|---|---|
| 본사 | OSPF Area 0 | Backbone Area |
| 대전 지사 | OSPF Area 3/4 | ABR Virtual Link로 Backbone 연결 |
| 부산 지사 | OSPF Area 2 + EIGRP 11 | EIGRP 재분배, VPN 터널 연동 |

OSPF랑 EIGRP가 맞닿는 경계에서 재분배(Redistribution)를 걸어, 두 프로토콜이 서로 경로 정보를 주고받게 했습니다. 이때 메트릭을 잘못 잡으면 엉뚱한 경로 루프나 서브최적 경로가 생기기 때문에, 값을 조정해가며 그런 일이 안 나도록 맞췄습니다.

### HSRP 게이트웨이 이중화

부산 지사 L3 스위치에는 HSRP(Hot Standby Router Protocol)를 구성했습니다. Active 장비가 죽으면 Standby가 Virtual IP·MAC을 그대로 넘겨받기 때문에, 클라이언트 쪽은 설정을 건드릴 필요 없이 몇 초 만에 통신이 살아납니다. Hello 패킷 간격이랑 Hold 타임을 줄여서 전환이 더 빨리 되도록 손봤습니다.

### 네트워크 보안

- **ACL** — 거점 간 트래픽을 허용 목록으로 명시 관리, 정의되지 않은 트래픽은 묵시적 차단
- **PAT** — 내부 사설 IP를 공인 IP로 변환하여 외부 통신 처리, 내부 구조 노출 최소화
- **EIGRP VPN 터널** — 지사 간 트래픽 암호화, 라우팅 정보 외부 노출 차단

## 사용 도구 및 환경

- **OS**: Rocky Linux 9 (부산 서버), Windows Server (대전 서버)
- **가상화**: VirtualBox
- **DB**: MariaDB (RAID 1+0 + Master-Slave Replication)
- **파일 공유**: Samba (SMB/CIFS)
- **라우팅 프로토콜**: OSPF (Area 0/2/3/4), EIGRP 11
- **게이트웨이 이중화**: HSRP
- **링크 이중화**: Ether Channel
- **VLAN 관리**: VTP Server/Client
- **보안**: SSH 공개키 인증, ACL, PAT, VPN, MariaDB IP 바인딩 제한
