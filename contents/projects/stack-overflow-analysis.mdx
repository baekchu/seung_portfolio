## 연구 개요

**「스택 오버플로우(Stack Overflow)의 발생 과정」**은 어셈블리 레벨에서 스택 버퍼 오버플로우가 발생하는 원리를, 명령어 동작 → 플래그 → 스택 프레임 → 실제 디스어셈블 코드 분석의 순서로 추적한 리버스 엔지니어링 학습 자료입니다.

| 항목 | 내용 |
|---|---|
| 분야 | 시스템 보안 / 리버스 엔지니어링 |
| 형태 | 개인 연구 자료 |
| 주요 키워드 | MOV/LEA, Carry/Overflow Flag, Stack Frame, Prologue/Epilogue, SFP/RET |
| 아키텍처 | x86 (IA-32), AT&T 어셈블리 문법 |

> 📄 **원문 문서**: [스택 오버플러우의 발생 과정.docx 내려받기](/documents/stack-overflow-analysis.docx)

## 1. MOV와 LEA 명령어 비교

스택 동작을 이해하려면 먼저 데이터를 옮기는 두 핵심 명령어의 차이를 알아야 합니다.

### MOV (move)
메모리·범용 레지스터·세그먼트 레지스터가 **참조하는 주소에 존재하는 데이터(값)** 를 그대로 전송합니다.

```asm
MOV EAX, ECX        ; ECX가 가진 '값'을 EAX에 복사
MOV AX, [BP+4]      ; [BP+4]를 하나의 '주소'로 보고, 그 주소에 들어있는 '값'을 AX에 로드
```
> 예) BP = `0x10000004`, `[BP+4]` = `0x10000008` 주소를 가리키고 그 주소의 값이 `1024`라면 → AX = `1024`

### LEA (Load Effective Address)
MOV처럼 제1피연산자를 목적지로 쓰는 것은 같지만, **제2피연산자(주소식)를 먼저 연산한 뒤 그 결과를 레지스터에 저장**한다는 점이 다릅니다. OPCODE의 ModR/M을 이용해 `SRC` 값을 계산해 `REG`에 넣습니다.

```asm
LEA EAX, [EBP + 1000]   ; EBP + 1000을 '계산'한 결과를 EAX에 저장
                        ;  → EBP=1000 이면 EAX = 2000
LEA AX, [BP+4]          ; [BP] 주소에 +4 '연산'을 적용한 값을 AX에 저장
                        ;  → BP=0x10000004 이면, 0x10000004 + 4 = 0x10000008 (값이 아닌 계산 결과)
```

### 핵심 차이

| 구분 | MOV | LEA |
|---|---|---|
| 가져오는 것 | 주소가 **가리키는 값** | 주소 **계산식의 결과** |
| 연산 가능 여부 | 옮기는 값에 연산 **불가** | 주소식에 덧셈·스케일 **연산 가능** |
| 메모리 접근 | 실제 메모리 참조 | 메모리 참조 없이 주소만 계산 |

> 정리 — **MOV는 값을 옮기기만** 하고, **LEA는 주소 계산을 수행**한다. 그래서 `LEA REG, 1234`와 `MOV EAX, 1234`는 결과가 같아 보여도 동작 원리가 다릅니다.

## 2. 캐리(Carry) 플래그와 오버플로우(Overflow) 플래그

버퍼 경계를 넘는 연산을 이해하기 위해 상태 레지스터의 두 플래그를 정리했습니다.

### 캐리 플래그
연산 결과의 MSB(최상위 비트)에서 **자리올림(carry)** 또는 **자리내림(borrow)** 이 발생하면 set 됩니다. 자리올림은 덧셈, 자리내림은 뺄셈에서 나타납니다.

```
1111 + 0001 = 0000   → Carry ON   (덧셈 자리올림 발생)
0000 - 0001 = 1111   → Carry ON   (뺄셈 자리내림 발생)
0111 + 0001 = 1000   → Carry OFF
1000 - 0001 = 0111   → Carry OFF
```

### 오버플로우 플래그
덧셈에서 **두 피연산자의 부호 비트가 같을 때**를 전제로, 결과의 부호 비트가 바뀌면 set 됩니다.
- 양수 + 양수 = 음수 → Overflow ON
- 음수 + 음수 = 양수 → Overflow ON
- 부호가 **다른** 두 수의 연산 → Overflow는 **절대 발생하지 않음**

### 플래그 상태 예측 vs 실제 (4비트 기준)

부호 있는 4비트의 표현 범위는 **7 ~ -8**입니다.

```
[예 1]  1000 + 1000 = (1)0000
        예측: 표현 범위 초과 → Overflow 일 것
        실제: Carry = 1, Overflow = 0
        이유: 자리올림은 발생했으나, 하위 4비트(0000)는
              올바른 값을 표현 → 오버플로우 아님

[예 2]  0111 + 0001 = 1000
        실제: Carry = 0, Overflow = 1
        이유: 최상단 비트 자리올림 없음 → Carry off
              양수(7) + 양수(1)의 결과가 음수(-8) → Overflow on
              올바른 결과 8은 4비트로 표현 불가
```

| 플래그 | set 조건 |
|---|---|
| **Carry** | 최상단 비트에서 자리올림(또는 자리내림) 발생 시 |
| **Overflow** | 최대 표현 숫자를 넘기거나, 같은 부호를 더한 결과가 다른 부호일 때 |

## 3. 스택 프레임(Stack Frame) 구조

함수가 호출되면 스택에는 다음과 같은 프레임이 형성됩니다. 스택은 **높은 주소 → 낮은 주소** 방향으로 자랍니다.

```
높은 주소
┌───────────────────────────┐
│  함수 인자 (arguments)       │   ← 8(%ebp), 12(%ebp) ...
├───────────────────────────┤
│  RET (return address)      │   ← 함수 종료 후 복귀할 주소
├───────────────────────────┤
│  SFP (saved EBP)           │   ← 이전 프레임 포인터  ← %ebp 가 여기를 가리킴
├───────────────────────────┤
│  지역변수 / 버퍼 buffer[]    │   ← -4(%ebp), -8(%ebp) ...  ← %esp
└───────────────────────────┘
낮은 주소        ▲ 버퍼에 넘치게 쓰면 위쪽(SFP·RET)을 덮어쓴다
```

| 용어 | 의미 |
|---|---|
| **EBP** | 함수의 기준점(base pointer). 지역변수·인자를 `±offset(%ebp)`로 접근 |
| **ESP** | 스택의 최상단(top) |
| **SFP** (Saved Frame Pointer) | 스택에 저장된 **이전 EBP** |
| **RET** (Return Address) | 함수 종료 시 점프할 **복귀 주소** |

핵심: 지역 버퍼는 EBP보다 **낮은 주소**에 있고, SFP·RET은 **높은 주소**에 있습니다. 따라서 버퍼에 할당량보다 많은 데이터를 쓰면 데이터가 위쪽으로 흘러 **SFP → RET 순서로 덮어써지고**, RET이 조작되면 공격자가 원하는 주소로 제어 흐름을 빼앗을 수 있습니다.

## 4. 함수 프롤로그와 에필로그

### 프롤로그 (Prologue) — 함수 시작 시 항상 수행
```asm
pushl %ebp          ; ① 이전 프레임 포인터(SFP)를 스택에 저장
movl  %esp, %ebp    ; ② 현재 ESP를 새 기준점 EBP로 설정
subl  $16, %esp     ; ③ 지역변수용 16바이트 스택 공간 할당
```

### 에필로그 (Epilogue) — 함수 종료 시
```asm
leave               ; movl %ebp,%esp + popl %ebp 와 동일 (프레임 해제)
ret                 ; 스택에 저장된 RET 주소로 점프하여 복귀
```

`pushl %ebp` + `movl %esp, %ebp`는 **모든 함수 시작 시 동일하게 수행되는 관용 패턴**입니다.

## 5. 전체 코드 디스어셈블 분석

`main()` → `add_func()` → `sub_func()` 호출 구조를 한 줄씩 분석했습니다.

### main() — 프롤로그와 지역변수 초기화
```asm
pushl %ebp
movl  %esp, %ebp
subl  $16, %esp           ; 지역변수 16바이트 할당
movl  $20, -4(%ebp)       ; a = 20  (ebp-4)
movl  $10, -8(%ebp)       ; b = 10  (ebp-8)
cmpl  $10, -4(%ebp)       ; a == 10 ?  → 거짓 (a=20)
```

### main() — 조건 분기 (if / else)
```asm
; if (a == 10)
pushl $.LC0              ; .string "a is 10" 주소 push
call  puts               ; puts("a is 10")
addl  $4, %esp           ; 인자 정리(4바이트)
jmp   .L3                ; if 블록 종료 후 점프

; else
pushl $.LC1              ; .string "a is not 10" 주소 push
call  puts               ; puts("a is not 10")
addl  $4, %esp           ; 인자 정리
```

### main() → add_func() 호출
```asm
movl  -8(%ebp), %eax     ; b 값을 eax로
pushl %eax               ; 인자 b push
movl  -4(%ebp), %eax     ; a 값을 eax로
pushl %eax               ; 인자 a push
call  add_func           ; add_func(a, b)
```

### add_func() — b = b + a
```asm
pushl %ebp               ; 프롤로그
movl  %esp, %ebp
movl  8(%ebp),  %eax     ; 인자 a (ebp+8) → eax
addl  %eax, 12(%ebp)     ; b(ebp+12) = b + a
movl  12(%ebp), %edx     ; 결과(b) → edx
movl  %edx, %eax         ; 반환값을 eax에 복사
jmp   .L4
leave                    ; 에필로그
ret                      ; 복귀
```

### main() — 반환값 저장 후 sub_func() 호출
```asm
addl  $8, %esp           ; 인자 2개(8바이트) 정리
movl  %eax, %eax
movl  %eax, -12(%ebp)    ; c = add_func 결과 (ebp-12)

movl  -8(%ebp), %eax     ; b
pushl %eax
movl  -4(%ebp), %eax     ; a
pushl %eax
call  sub_func           ; sub_func(a, b)
```

### sub_func() — a = a - b
```asm
pushl %ebp               ; 프롤로그
movl  %esp, %ebp
movl  12(%ebp), %eax     ; 인자 b (ebp+12) → eax
subl  %eax, 8(%ebp)      ; a(ebp+8) = a - b
movl  8(%ebp), %edx      ; 결과(a) → edx
movl  %edx, %eax         ; 반환값을 eax에 복사
jmp   .L5
leave
ret
```

### main() — 종료
```asm
addl  $8, %eax
movl  %eax, %eax
movl  %eax, -16(%ebp)    ; d = sub_func 결과 (ebp-16)
xor   %eax, %eax         ; eax = 0 (return 0). XOR: 두 비트가 다르면 1, 같으면 0
jmp   .L1
leave
ret                      ; 프로그램 종료
```

> `±offset(%ebp)`로 지역변수(`-4`, `-8`, `-12`, `-16`)와 인자(`+8`, `+12`)에 접근하는 패턴을 통해, 컴파일러가 EBP 기준으로 스택 프레임을 어떻게 배치하는지 확인할 수 있습니다.

## 6. 스택 오버플로우가 발생하는 원리

위 분석을 종합하면, 스택 오버플로우는 다음 과정으로 발생합니다.

1. 함수 프롤로그가 스택에 `버퍼 → SFP → RET` 순으로 프레임을 만든다.
2. `gets()`·`strcpy()`처럼 **경계 검사 없는 입력**으로 버퍼에 할당량보다 많은 데이터를 쓴다.
3. 데이터가 버퍼를 넘어 **낮은 주소 → 높은 주소** 방향으로 흘러 **SFP를 지나 RET까지 덮어쓴다**.
4. 함수가 `ret`로 복귀할 때, 오염된 RET이 가리키는 **공격자가 지정한 주소(쉘코드 등)** 로 점프한다.
5. 결과적으로 프로그램의 **제어 흐름이 탈취(control-flow hijack)** 된다.

```
정상:   [ buffer ][ SFP ][ RET=원래복귀주소 ] → 정상 복귀
공격:   [ AAAA…AAAA  …  AAAA ][ 0x41414141 ] → RET 오염 → 임의 주소로 점프
```

## 7. 결론

스택 오버플로우는 단순한 버그가 아니라 **MOV/LEA의 데이터·주소 처리, 플래그 set 조건, EBP/ESP 기반 스택 프레임, 프롤로그/에필로그 동작**이 맞물려 일어나는 현상입니다. 어셈블리 레벨에서 함수 호출 규약과 스택 레이아웃을 정확히 이해하면, 버퍼 경계를 넘는 쓰기가 어떻게 SFP·RET을 오염시켜 제어 흐름을 탈취하는지 명확히 파악할 수 있고, 이는 스택 카나리·DEP·ASLR 같은 방어 기법의 필요성으로 이어집니다.
